# Epic 14: Security and Reliability

Обязательное условие для всех задач: строгое соблюдение [Project Rules](./00-project-rules.md).

## E14-T01 Применить least-privilege для Tauri capabilities

**Контекст:** Desktop-приложение с импортом файлов требует минимально необходимых разрешений.

**Зависимости:** E01-T02, E12-T04.

**Критерии приемки:**
1. Конфиг capabilities ограничен только используемыми API.
2. Удалены/запрещены неиспользуемые разрешения.
3. Добавлен security checklist для ревью capability-изменений.

**Условия:**
- Любое расширение capability должно быть отдельно обосновано в PR.

## E14-T02 Настроить строгий CSP и запрет нежелательных remote ресурсов

**Контекст:** CSP снижает риски внедрения вредоносного кода.

**Зависимости:** E14-T01.

**Критерии приемки:**
1. CSP задан явно для production-конфига.
2. Запрещены ненужные remote script/style источники.
3. Приложение успешно запускается и работает под новым CSP.

**Условия:**
- Нельзя отключать CSP ради быстрого обхода ошибок.

## E14-T03 Усилить валидацию IPC payload и файловых путей

**Контекст:** Главные точки входа данных — IPC и импорт файлов.

**Зависимости:** E02-T02, E12-T04.

**Критерии приемки:**
1. Все IPC-команды валидируют payload на входе.
2. Добавлена нормализация и проверка файловых путей.
3. Невалидный payload/путь возвращает безопасную ошибку без паники.

**Условия:**
- Сообщения об ошибках не должны утекать внутренние детали runtime.

## E14-T04 Реализовать recovery-механизмы и smoke tests отказоустойчивости

**Контекст:** MVP должен переживать ошибки парсинга и сбои симуляции без потери пользовательских данных.

**Зависимости:** E14-T03, E10-T04.

**Критерии приемки:**
1. После сбоя симуляции UI возвращается в консистентное состояние.
2. После неуспешного импорта приложение продолжает работу без перезапуска.
3. Добавлен smoke-набор тестов на основные failure-сценарии.

**Условия:**
- Восстановление не должно молча удалять пользовательские данные.
