# Epic 02: Architecture and Contracts

Обязательное условие для всех задач: строгое соблюдение [Project Rules](./00-project-rules.md).

## E02-T01 Зафиксировать архитектурные границы и ответственность модулей

**Контекст:** До функциональной реализации нужно задать стабильные boundaries между UI, domain, simulation, storage и import.

**Зависимости:** E01-T06.

**Критерии приемки:**
1. Создан ADR-документ с описанием слоев и направлений зависимостей.
2. Для каждого слоя задан список разрешенных импортов/зависимостей.
3. Определены anti-patterns (например, прямой доступ UI к storage).

**Условия:**
- ADR должен быть кратким и применимым для code review.
- Терминология должна совпадать с PRD.

## E02-T02 Определить IPC-контракты между React и Rust

**Контекст:** Основной риск Tauri-приложений — неявные, невалидируемые payload между frontend и backend.

**Зависимости:** E02-T01.

**Критерии приемки:**
1. Определен каталог DTO/command contracts.
2. Для MVP-команд создана таблица: command name, input schema, output schema, error schema.
3. Внедрена базовая runtime-валидация входов на Rust-стороне.

**Условия:**
- Схемы должны быть версионируемыми (`v1/*` namespace или эквивалент).

## E02-T03 Внедрить единый error model и логирование

**Контекст:** Для отладки симуляции и импорта нужна единая структура ошибок и логов.

**Зависимости:** E02-T02.

**Критерии приемки:**
1. Есть типизированные категории ошибок (`validation`, `io`, `simulation`, `import`, `internal`).
2. Ошибки backend корректно маппятся в user-facing сообщения frontend.
3. Добавлен correlation/request id для трассировки command flow.

**Условия:**
- Не раскрывать внутренние stack traces пользователю в production режиме.

## E02-T04 Добавить feature flags для модулей MVP

**Контекст:** Нужно безопасно включать/отключать подсистемы на этапе интеграции.

**Зависимости:** E02-T03.

**Критерии приемки:**
1. Реализован механизм feature flags для frontend и backend.
2. Флагами покрыты минимум: `simulation`, `import_export`, `advanced_precision`.
3. Документировано поведение приложения при выключенном модуле.

**Условия:**
- Нельзя допустить silent-failure при выключении функции.
